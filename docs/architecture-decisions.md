# Architecture Decisions

Key design decisions, the reasoning behind them, and the trade-offs accepted.

---

## Import strategy: Takeout + Picker instead of Google Photos Library API

**Decision:** Replace the Google Photos Library API with a hybrid of Google Takeout (bulk) and the Google Photos Picker API (incremental).

**Why:** Google deprecated the Library API for new OAuth clients in early 2025. The replacement ecosystem splits into two tools with different purposes:

- **Takeout** is better for importing your entire existing library in one go. No auth required — it's a folder on disk. JSON sidecars carry the original EXIF timestamps and titles that Google preserved, making it the most accurate source for historical data.
- **Picker API** is better for incremental "top-up" imports of recent photos. It requires OAuth but only accesses photos the user explicitly selects in a browser UI, which is a better privacy model than requesting read access to the entire library.

**Trade-off accepted:** The Picker API's `createTime` is the server upload time, not the EXIF capture time. This means cross-source deduplication between a Takeout import and a Picker import of the same photo is best-effort — the fingerprints may not match if the photo was taken and uploaded at different times. Within a single source (reimporting the same Takeout archive, or reselecting the same photos in Picker) deduplication works reliably.

---

## Deduplication: fingerprint instead of Google photo ID

**Decision:** Dedup key is `{unix_timestamp}_{filename}`, stored as a UNIQUE index in SQLite.

**Why:** Google's photo IDs are source-specific. A photo imported via Takeout gets a locally-generated UUID; the same photo from the Picker API has a Google-assigned ID. There is no stable cross-source identifier available. Using a content-derived fingerprint (original capture timestamp + original filename) gives reasonable dedup across sources without requiring a canonical ID.

**Trade-off accepted:** If two different photos happen to have the same filename and were taken in the same second (e.g. burst mode), the second will be silently skipped as a duplicate. This is an acceptable edge case for a personal library.

---

## Localhost OAuth redirect server instead of OOB flow

**Decision:** `start_auth_flow` binds a random-port TCP listener on `127.0.0.1`, registers it as the OAuth redirect URI, and blocks until the browser delivers the `?code=` parameter.

**Why:** Google deprecated the "copy-paste code" OOB flow (`urn:ietf:wg:oauth:2.0:oob`) for installed apps. The recommended replacement for desktop apps is a localhost redirect. The random port avoids conflicts with other running services. The single blocking command is simpler to reason about than the old two-step `get_auth_url` + `exchange_auth_code` pair.

---

## Embedding model: `gemini-embedding-2-preview` at 1536 dimensions

**Decision:** Use `gemini-embedding-2-preview` as a single multimodal model for both image indexing and text query embedding, at 1536 dimensions.

**Why multimodal rather than two models:** Both image thumbnails and text queries land in the same embedding space, so cosine similarity scores are meaningful across them. Using a single model eliminates the alignment problem that arises when you embed images with one model and queries with another.

**Why 1536 instead of 3072 (the default):** The model supports flexible output dimensions (128–3072) with 768, 1536, and 3072 as recommended breakpoints. At 1536 dimensions, storage is roughly half that of 3072 (~17 MB per 1,000 photos in SQLite JSON text vs ~33 MB) with negligible quality loss for a personal photo library. The reduced dimensions are a trained truncation, not a naive slice.

**Previous model:** `gemini-embedding-exp-03-07` was used initially but was retired by Google in early 2025. Any embeddings generated with the old model are incompatible with the new one (different dimensions and vector space) and must be regenerated by clearing the `embeddings` table and re-running indexing.

---

## AI descriptions: stored at index time, independent of embedding

**Decision:** At index time, generate a natural language description of each photo using `gemini-2.5-flash` and store it in `photos.description`. This is a separate step from embedding — the image bytes go directly to `gemini-embedding-2-preview` unchanged.

**Why store descriptions:**
- Displayed in the Inspector panel so users can see what the AI understood about a photo
- Enables future full-text search (SQLite FTS5) as a complement to vector search
- Makes search results interpretable — you can see why a photo matched

**Why not embed the description instead of the image:** The multimodal embedding of the raw image captures visual information (colour, composition, spatial relationships) that a text description may not preserve. Descriptions are better at labelling named entities and scene types; image embeddings are better at visual similarity. Keeping both gives the best of both.

**Why not dual embeddings (image + description):** Adds significant complexity — two vectors per photo, combined scoring logic, 2× storage. The marginal search quality improvement doesn't justify it at this stage.

---

## People search: face anchoring, not semantic search

**Decision:** People search (Stage 3) will use face similarity search against user-labelled anchors, not semantic search.

**Why semantic search won't work for people:** `gemini-embedding-2-preview` has no knowledge of who private individuals are. It can embed a photo of Ada Lovelace and produce a vector near "woman with dark hair smiling outdoors", but searching for "Ada" produces no signal — there is nothing in the vector space that maps the name "Ada" to that person's face.

**Planned approach:** The user selects one photo of a person as an anchor and names them. A face crop is extracted, embedded with a face-specific model, and stored in the `people` table. Searching for a person performs nearest-neighbour lookup in face embedding space, not in the general semantic space. The two search modes (semantic + face) coexist.

---

## Thumbnail storage: generated at import time, referenced by ID

**Decision:** Thumbnails are generated as 512px JPEG files at import time and stored in `{app_data}/thumbnails/{photo_id}.jpg`. The path is computed from the photo ID at query time — it is never stored in the database.

**Why not store the path:** The app data directory can change between installations or OS versions. Computing the path from the stable photo ID makes the database portable and avoids stale path entries.

**Why not store base64 in the database:** SQLite is not a blob store. Embedding thumbnail data in the DB would balloon its size and slow down queries that scan the photos table.

**Takeout vs Picker originals:** Takeout originals are referenced in-place from wherever the user placed the Takeout folder — the `local_path` column stores the absolute path. Picker originals are downloaded to `{app_data}/photos/`. This means Takeout imports survive a DB wipe (originals are on disk), but moving or deleting the Takeout folder breaks the reference.

---

## Embedding storage: JSON text in SQLite

**Decision:** Float32 vectors are serialised as JSON arrays (`[0.123, -0.456, ...]`) and stored in the `embeddings` table as TEXT.

**Why not BLOB:** Storing as binary `f32` would be ~5× smaller (~6 KB vs ~30 KB at 1536 dims) and faster to read, but requires careful endianness handling and makes the database harder to inspect. For the current scale (tens of thousands of photos) the size difference is manageable. If the library grows to hundreds of thousands of photos this is worth revisiting.

**Security note:** The SQLite database is stored in the OS app data directory and is not encrypted. Embedding vectors cannot reconstruct the original image, but filenames and AI-generated descriptions are stored in plaintext. Anyone with local access to the user account can read the database. This is standard for desktop apps of this type; SQLCipher would be required for at-rest encryption.
